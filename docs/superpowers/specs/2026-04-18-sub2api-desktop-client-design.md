# Sub2API 桌面客户端设计文档

- 日期：2026-04-18
- 状态：已在对话中完成设计确认，等待用户审阅文档
- 范围：V1 桌面客户端 + `sub2api` 最小增量服务端改造

## 背景

目标是为 `sub2api` 构建一个面向终端用户的独立跨平台桌面客户端。客户端不展示 API Key、Base URL 或其他原始连接信息，而是直接启动本机已安装的 `Codex Desktop` 和 `Codex CLI`，支持两种模式：

- `官方模式`：启动用户原本已经配置好的官方 Codex 环境
- `平台模式`：通过受管桌面会话将 Codex 接到用户自己的 `sub2api` 服务

这套设计必须同时满足以下要求：

- 官方环境与平台代理环境完全隔离
- 首版实现和后期维护成本可控
- 不破坏 `sub2api` 现有网页端和现有 API 行为

## 目标

- 提供一个面向终端用户的全定制跨平台桌面客户端
- 首发优先保证 `Windows` 和 `macOS`
- 首版同时支持 `Codex Desktop` 和 `Codex CLI`
- 保证 `官方模式` 与 `平台模式` 完全隔离
- 在 UI 中隐藏 API Key、Base URL 与原始网关凭据
- 复用 `sub2api` 现有的登录、注册、余额、用量、CDK、公告等能力
- 仅对 `sub2api` 增加最小必要的桌面会话能力
- 启动切换保持为“会话级切换”，不改用户系统级默认状态

## 非目标

- V1 不做客户端内支付或内嵌收银台
- V1 不做 `Codex Desktop` / `Codex CLI` 自动下载与安装
- 不管理官方账号的登录、登出、多账号切换
- V1 不支持多站点或多后端切换
- 不在桌面客户端中展示、复制或管理 API Key
- 不重写现有网页端流程，也不改变现有 API Key 用户的接入方式

## 已确认的产品决策

以下约束已经在设计讨论中确认：

- 形态：跨平台桌面客户端
- 受众：终端用户
- 连接方式：固定连接单一 `sub2api` 站点
- UI 方案：全定制客户端界面，不使用 WebView 壳
- 信息架构：以仪表盘为主的首页结构
- 认证范围：登录、注册、忘记密码、邮箱验证码
- 账户页面：余额、用量、CDK 兑换、充值说明页
- API Key 管理：从 V1 界面中完全移除
- 启动对象：`Codex Desktop` 与 `Codex CLI`
- 切换方式：仅支持会话级切换
- 官方模式：只负责启动本机已登录的官方 Codex，不管理官方账号状态
- 安装前提：V1 依赖用户本机已安装 Codex 产品
- 数据隔离：官方历史与平台代理历史必须完全隔离
- 服务端改造：允许小改，但必须是增量式，不能破坏现有网页端和现有 API 行为

## 推荐技术方向

### 客户端运行时

推荐的 V1 技术栈：

- `Rust`
- `Slint`

推荐理由：

- 这个项目的核心难点是本机进程拉起、临时注入、Profile 隔离、异常恢复，天然更偏桌面控制器而不是网页壳客户端
- `Rust` 更适合处理子进程管理、文件系统隔离、临时目录生命周期和恢复清理这类强本机能力
- `Slint` 更贴近原生桌面 UI 方向，适合做长期的桌面产品形态，而不是把网页界面塞进一个桌面壳
- 首版虽然开发速度可能慢于 Web 技术栈，但长期看更符合“强接管桌面客户端”的目标

附加约束：

- 商业发布前需要明确 `Slint` 的许可使用方式，确认是否接受免费许可下的署名要求，或者是否需要商业许可

这是一条推荐技术路线，也是当前已确认的实现方向。如果后续因为授权、团队能力或交付节奏原因需要回退到其他桌面运行时，应单独修订本设计文档。

## 系统架构

V1 建议拆成 7 个客户端模块，加上 1 个服务端增量能力切片。

### 1. 账户中心

职责：

- 登录
- 注册
- 忘记密码
- 邮箱验证码
- 用户信息摘要
- 余额展示
- 用量展示
- CDK 兑换
- 充值说明内容展示

主要依赖现有服务端接口：

- `/api/v1/auth/*`
- `/api/v1/auth/me`
- `/api/v1/user/*`
- `/api/v1/usage/*`
- `/api/v1/redeem/*`
- 公告与订阅摘要相关接口

### 2. 本机探测

职责：

- 探测本机是否安装 `Codex Desktop`
- 探测本机是否安装 `Codex CLI`
- 识别可执行文件路径
- 确认目标是否可启动
- 将探测结果反馈给界面

### 3. 模式切换器

职责：

- 记录用户当前偏好的默认启动模式：`官方模式` 或 `平台模式`
- 仅对从本客户端发起的启动生效
- 不改系统全局默认状态

### 4. Profile 管理器

职责：

- 维护平台代理模式下的受管持久 Profile
- 分别维护 `platform-desktop` 与 `platform-cli`
- 保留平台代理历史与状态
- 永远不修改用户原本的官方 Profile

### 5. 启动编排器

职责：

- 为每次启动准备临时注入材料
- 启动 `Codex Desktop` 和 `Codex CLI`
- 处理桌面会话续期
- 监听子进程退出
- 触发清理与恢复逻辑

### 6. 安全存储适配层

职责：

- 将桌面客户端自己的登录态与敏感信息存放到系统安全存储中
- 避免把长期可用的网关凭据写入受管 Profile
- 避免敏感值进入日志、剪贴板或普通 UI 展示流程

### 7. 服务端适配层

职责：

- 调用现有 `sub2api` 账户相关接口
- 调用新增的桌面会话接口
- 将服务端返回状态统一整理为桌面客户端可消费的数据

### 8. 服务端桌面会话切片

职责：

- 为已登录用户签发桌面运行会话
- 校验桌面启动使用的短期凭据
- 处理桌面会话刷新与撤销
- 让桌面凭据与普通 API Key 在语义上彻底分离

## 服务端增量设计

服务端改造必须是增量式的，不影响现有网页端，也不影响现有 API Key 用户。

### 新增路由命名空间

建议新增一个独立路由组：

- `/api/v1/desktop/*`

这样既贴合 `sub2api` 现有路由风格，又能让桌面客户端专属能力和现有认证、Key 管理接口清晰隔离。

### 新增接口

#### `POST /api/v1/desktop/sessions`

用途：

- 为当前已认证用户创建一个可用于桌面启动的运行会话

请求字段：

- `target`：`desktop` 或 `cli`
- `device_id`
- `device_name`
- `client_version`

返回字段：

- `session_id`
- `expires_at`
- `gateway_base_url`
- `runtime_token`
- `refresh_after`
- `profile_key`

说明：

- `runtime_token` 不是普通 API Key
- `profile_key` 用于让客户端把本次启动绑定到正确的受管 Profile 家族，而不暴露内部凭据语义

#### `POST /api/v1/desktop/sessions/:id/refresh`

用途：

- 在桌面会话仍然活跃时，为其续期

行为：

- 返回新的过期时间
- 如有需要，可返回轮换后的 `runtime_token`

#### `DELETE /api/v1/desktop/sessions/:id`

用途：

- 在正常退出时主动关闭桌面会话

行为要求：

- 幂等
- 可重复调用
- 如果客户端异常退出，仍由服务端过期策略兜底

### 新增桌面兼容网关入口

建议新增一条专供桌面模式流量使用的兼容入口，例如：

- `/api/desktop/v1`

这条路径只校验 `runtime_token`，而不是普通用户 API Key。服务端内部再将其映射回当前桌面会话、用户、计费和策略上下文。

这样可以同时保证：

- 桌面运行凭据不能被当作普通 API Key 使用
- 现有 API 使用方的行为不受影响

### 建议新增的数据模型

建议新增一个桌面会话数据模型，至少包含：

- `id`
- `user_id`
- `device_id`
- `target`
- `status`
- `runtime_token_hash`
- `expires_at`
- `last_seen_at`
- `created_at`
- `revoked_at`

建议但非强制字段：

- `client_version`
- `device_name`
- `profile_key`
- `last_ip`

## 凭据时效策略

桌面运行凭据必须是“短期的，但不能脆弱”。

建议策略：

- `runtime_token` 默认有效期：`12 小时`
- 自动检查续期：每 `30 分钟` 一次
- 提前续期阈值：剩余时间少于 `2 小时`
- 续期失败缓冲期：`15-30 分钟`
- 正常退出时主动撤销会话
- 长时间失联或异常退出时由服务端自然过期

这样可以兼顾两件事：

- 不让凭据长期裸露在本机
- 不让用户因为时效过短而在正常长会话中频繁掉线

## 本地数据模型与 Profile 隔离

客户端本地分成 3 层存储。

### 1. App State

客户端自有的持久状态：

- 当前登录用户摘要
- 默认模式偏好
- 已探测到的安装路径
- 设备标识
- 最近一次启动记录
- 异常退出恢复标记

### 2. Managed Profiles

平台代理模式下的持久工作状态：

- `platform-desktop`
- `platform-cli`

这些目录用来保存平台代理模式的历史和本地状态，但不能保存长期可用的原始网关凭据。

### 3. Runtime Sessions

每次启动的临时运行状态：

- `session_id`
- 临时运行凭据
- 本次启动的连接覆盖配置
- 会话续期所需的记录信息

这些目录在正常退出时清理；如果上次异常退出，则在下次启动时优先清理。

### 隔离规则

- 官方 Profile 永远不被客户端修改
- `platform-desktop` 与 `platform-cli` 各自隔离
- 官方历史与平台代理历史彻底隔离
- 临时启动凭据不能沉淀进长期受管 Profile

## 启动流程

### 官方模式

流程：

1. 客户端检测本机是否存在目标 Codex 程序
2. 直接使用用户现有官方环境启动目标程序
3. 不创建任何 `sub2api` 桌面会话
4. 不修改任何官方 Profile 文件

结果：

- 官方登录状态仍然完全由用户自己管理
- 官方历史继续保存在原本官方环境所在的位置

### `Codex Desktop` 的平台代理模式

流程：

1. 客户端确认用户已登录 `sub2api`
2. 调用 `POST /api/v1/desktop/sessions`
3. 服务端返回短期桌面会话
4. 客户端针对 `platform-desktop` 这套受管 Profile 注入本次启动所需的连接覆盖
5. 启动 `Codex Desktop`
6. 进程存活期间按策略自动续期
7. 退出时调用 `DELETE /api/v1/desktop/sessions/:id` 并清理运行时残留

### `Codex CLI` 的平台代理模式

流程：

1. 客户端确认用户已登录 `sub2api`
2. 调用 `POST /api/v1/desktop/sessions`
3. 服务端返回短期桌面会话
4. 客户端针对 `platform-cli` 这套受管 Profile 注入本次启动所需的连接覆盖
5. 启动 `Codex CLI`
6. 进程存活期间按策略自动续期
7. 退出时关闭桌面会话并清理运行时残留

### 下次启动时的恢复流程

如果上一次是客户端崩溃、系统异常或 Codex 进程被强杀：

1. 扫描未清理的运行时会话目录
2. 尝试撤销对应的服务端桌面会话
3. 删除残留的临时注入文件
4. 保留平台代理的长期历史

这样即使上次退出不干净，受到影响的也只是“上次会话的临时连接材料”，不会破坏平台代理历史，更不会污染官方环境。

V1 采用以仪表盘为主的布局，不再提供任何 API Key 视角的页面。

### 包含的页面

- 登录
- 注册
- 忘记密码
- 邮箱验证码
- 仪表盘首页
- `Codex Desktop` / `Codex CLI` 启动面板
- 官方模式 / 平台模式切换器
- 余额摘要
- 用量摘要
- CDK 兑换
- 充值说明页

### 不包含的页面

## 错误处理

错误处理按 4 类组织。

### 1. 桌面会话创建失败

示例：

- 用户登录态失效
- 服务端暂时不可用
- 设备被拒绝

处理方式：

- 平台模式启动被拦截
- 明确提示“平台连接当前不可用”
- 官方模式仍可继续启动

### 2. 本机探测失败

示例：

- 未安装 `Codex Desktop`
- `Codex CLI` 路径失效
- 可执行文件无法正常启动

处理方式：

- 不显示伪启动成功状态
- 只将缺失目标标记为不可用
- 另一个已安装目标仍然可用

### 3. 运行中续期失败

处理方式：

- 在缓冲窗口内后台重试
- 不立即终止当前进程
- 仅在运行凭据真正过期后，才要求用户重新启动平台代理模式

### 4. 异常退出

处理方式：

- 记录异常退出状态
- 下次启动时优先执行恢复清理
- 保留平台代理历史

## 降级策略

降级目标是：平台坏了，官方还能用；一个启动目标坏了，另一个还能用。

- 如果 `sub2api` 桌面会话能力暂时不可用，官方模式仍可正常启动
- 如果只安装了 `Codex Desktop`，则只开放 Desktop 入口
- 如果只安装了 `Codex CLI`，则只开放 CLI 入口
- 如果平台模式启动失败，登录、余额、用量、CDK、充值说明等账户功能仍然可用

## 安全规则

以下约束是 V1 的硬性规则：

- 不在客户端中展示普通 API Key
- 不在日志中记录原始网关凭据
- 不提供可复制的原始可用凭据
- 不把长期可用网关凭据写入受管 Profile
- 不通过“修改官方配置再恢复”的方式接管官方环境
- 桌面运行凭据与普通 API Key 必须在语义上彻底分离

## V1 范围边界

V1 包含：

- 固定单一 `sub2api` 后端
- 以 `Windows` 与 `macOS` 为首发优先平台
- 面向终端用户的账户流程
- 登录、注册、忘记密码、邮箱验证码
- 余额、用量、CDK、充值说明
- 官方模式 / 平台模式切换
- `Codex Desktop` 与 `Codex CLI` 启动支持
- 受管平台代理 Profile + 运行时会话注入
- `sub2api` 服务端新增最小桌面会话支持

V1 不包含：

- 客户端内支付
- 官方账号管理
- 自动下载与安装
- 多后端白牌切换
- 系统级全局模式切换
- 桌面客户端内的 API Key 管理
- 本地常驻桥接代理

## 验收标准

满足以下条件即可视为首版闭环：

- 用户可在桌面客户端完成注册、登录、重置密码、邮箱验证
- 用户可查看余额与用量，可进行 CDK 兑换，并查看充值说明
- 客户端可稳定识别本机已安装的 `Codex Desktop` 和 `Codex CLI`
- 用户可在 `官方模式` 下启动已安装目标
- 用户可在 `平台模式` 下启动已安装目标
- 官方模式不会修改用户原有官方环境
- 平台模式通过桌面会话启动，且不会暴露普通 API Key
- 官方历史与平台代理历史完全隔离
- 异常退出后可自动清理上次遗留的运行时会话残留
- 即使 `sub2api` 桌面会话能力异常，官方模式仍然可作为降级路径

## 后续实现阶段的测试策略

实现计划至少要覆盖以下测试层级：

- `/api/v1/desktop/*` 的服务端契约测试
